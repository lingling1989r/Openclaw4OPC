### 互联网补充：OpenClaw 的“框架”（你第一章要讲的系统结构）

如果你第一章要讲“OpenClaw 的框架”，官方有两段内容非常适合做“框架页”：

1. **一句话定义（产品是什么）**
   - README / FAQ 都强调：OpenClaw 是运行在你设备上的 personal AI assistant，**Gateway 是 always-on control plane，assistant 才是产品**。
   - 这句话能帮你把它和“聊天机器人 / 套壳”区分开。

2. **框架结构（怎么运行）**（架构视角）
   - 一个长期运行的 **Gateway（daemon）** 统一管理 messaging surfaces（WhatsApp/Telegram/Discord/iMessage/...）
   - 控制面客户端（CLI / Web UI / macOS app / automations）通过 **WebSocket** 连到 Gateway
   - **Nodes（iOS/Android/macOS/headless）** 也通过 WebSocket 连到同一个 Gateway，提供设备能力（camera/screen/location/canvas 等）

你可以把它总结成一张“框架四件套”幻灯片：

- **Channel（入口）**：用户在哪儿发消息
- **Gateway（中枢）**：路由、会话、权限、工具调度
- **Skills/Tools（能力）**：能干什么活、怎么干
- **Workspace/Memory（沉淀）**：怎么把能力变成复利

> - Gateway Architecture（组件与数据流）：https://docs.openclaw.ai/concepts/architecture.md

### 互联网补充：什么叫“channel / 最后一公里”？

把“龙虾”当作一个 Agent 体系理解时，**channel** 本质上是：

- **用户入口（distribution surface）**：用户在哪儿发指令，Agent 就在哪儿可用
- **触达与回传通道**：不是在电脑 IDE 里才能用，而是随时随地“发消息→拿结果”

这类设计的价值在于把复杂工具“带出桌面”，把 AI 的使用门槛压到“会发消息就会用”。

OpenClaw 的官方描述也非常贴近这个定义：一个 **self-hosted gateway** 把 WhatsApp / Telegram / Discord / iMessage 等聊天应用接到 agent 上——也就是说，“聊天应用本身”就是 channel；gateway 负责把消息路由到 agent 并把结果返回给用户。（参考：OpenClaw Features、Gateway Architecture）

- OpenClaw Features：强调“Multi-channel gateway”（一个 Gateway 对接多个聊天渠道）以及媒体/移动节点能力。
- Gateway Architecture：强调 Gateway 作为长期运行的中心组件，统一承载消息面（messaging surfaces）并通过 WebSocket 接入客户端与节点。

> 引用：
> - https://docs.openclaw.ai/concepts/features.md
> - https://docs.openclaw.ai/concepts/architecture.md

### 可直接复用的“龙虾框架（定义版）”

你可以用一句话把第一章讲透：

> **龙虾 = 有明确入口（channel）+ 可执行工具（skills）+ 可持续进化（memory/workspace）+ 可协作扩展（multi-agent）的生产力系统。**

## 02｜第二章节：龙虾能干什么？

### 原始要点

- **展示原则：不要只秀基础能力**
  - 建议展示“真正能干活”的能力，而不只是基础 skill
- **Skill 示例（可按场景挑选展示）**
  - 网页搜索、信息抓取等通用能力
  - **自媒体方向（我自己的尝试）**
    - 直接发布公众号
    - 重做选题
    - 写小红书内容（包括**小红书配图也能直接生成**）
  - **业务/任务方向**
    - 做饰品类任务（作为案例：从需求到交付的协作）
    - 处理各种任务流
    - 协调与组织多人团队：任务规划、汇总、复盘等

### 互联网补充：Skill 的“产品化定义”

OpenClaw 文档对 Skills 的定义比较清晰：Skills 是兼容 AgentSkills 规范的技能文件夹（通常包含 `SKILL.md`），用于**教会 agent 如何使用工具**；并且 Skills 有明确的加载位置与优先级（bundled / managed / workspace），可以按环境与配置进行 gating。（参考：Skills 文档）

这意味着你在第二章不只是展示“会做什么”，而是在展示一套“可交付能力”的封装方式：

- 能力被标准化（有说明书、有入口、有边界、有依赖条件）
- 能力可复用、可组合、可迭代
- 能力可以被“卖”（收费的前提是可被复制的交付）

> 引用：
> - https://docs.openclaw.ai/tools/skills.md

### 建议展示方式：从“点技能”升级到“端到端结果”

建议用 **3 层展示结构**（观众理解最快）：

1. **单点技能（1 分钟）**：搜索/转写/生成
2. **小工作流（3 分钟）**：输入→处理→输出（例如：转写→清理口癖→摘要→发布）
3. **端到端交付（5 分钟）**：带一个真实任务，从需求到最终交付（含失败重试与边界条件）

这样观众不会停留在“酷炫”，而是能直接对标自己的工作流。

## 03｜第三章节：如何养成这样的龙虾？

### 原始要点

- 前面 1～2 章的训练、练习各种 skill 的过程，本质上就是“养成龙虾”的过程
- **养龙虾 = 培养一个超级个体 / 超级 Agent**
- 培养超级 Agent 的两种方法：
  1. **简单养的方式**：直接告诉它要做什么（例如：让它把配图加上）
  2. **服务器端养**（更根本的方式）：在服务端进行能力沉淀与定制化升级（更推荐长期采用）

### 互联网补充：为什么“写进文件”才算养成？（Memory / Workspace）

OpenClaw 的 Memory 文档强调了一个关键点：

- Memory 是**工作区里的纯 Markdown 文件**；文件才是事实来源（source of truth）
- 模型本身不会永久记住任何东西，除非你把它写到 `memory/YYYY-MM-DD.md` 或 `MEMORY.md`

这和“养成”的本质非常一致：

- 你不是在“调教一次对话”，而是在**积累一套可复用的知识与 SOP**
- 能沉淀到文件里的东西，才会在后续每次任务里持续复利

> 引用：
> - https://docs.openclaw.ai/concepts/memory.md

### 一套更可操作的“养龙虾路线图”（从新手到可商用）

- **Level 1：能稳定复现（稳定性）**
  - 同一个任务，10 次里至少 8 次输出可用
  - 关键动作：把输入模板、验收标准、失败重试写成 SOP
- **Level 2：能组合技能（组合性）**
  - 把 3～5 个技能串成工作流（带中间产物与校验点）
- **Level 3：能交付结果（交付性）**
  - 输出不只是“文本/图片”，而是能直接用于发布/上线/发给客户的“交付包”
- **Level 4：能规模化（可复制）**
  - 把过程抽象为可售卖的标准化服务 / 产品

## 04｜组织变革：从超级个体到一人公司

### 原始要点

- 从“超级个体”进化到“一人公司”，意味着从个体户走向组织化
- 可以引出两个问题：
  - 为什么历史上会出现公司组织这种形态？
  - 常见组织架构有哪些？
- **一人公司的核心：配备多个 Agent**
  - 多 Agent 的组织方式也有门道，可展开为：
    - 按业务划分
    - 按技能划分
    - 按业务 + 技能混合划分

### 互联网补充：Multi-Agent 的“隔离”是组织成立的前提

OpenClaw 的 Multi-Agent Routing 文档把“一人公司”的工程前提讲得很清楚：

- 一个 agent 不只是“一个提示词”，而是一个完整的隔离单元：
  - 独立 workspace
  - 独立 state / auth profiles
  - 独立 sessions 存储
- 多 agent 并行存在时，需要明确的路由与绑定（bindings）来分流消息

换句话说：**当 agent 具备“隔离的职责边界 + 独立的上下文与记忆”时，你才真的拥有一个可管理的团队。**

> 引用：
> - https://docs.openclaw.ai/concepts/multi-agent.md

### 可直接拿来讲的“组织架构 3 套模板”

1. **按业务线（更像公司）**：增长 / 内容 / 客服 / 研发 / 财务
2. **按能力栈（更像工厂）**：写作 / 设计 / 数据 / 自动化 / 分发
3. **混合（最实用）**：
   - 业务 Agent 负责目标与验收（比如“增长”）
   - 能力 Agent 负责生产（比如“内容生产”“视频剪辑”“投放文案”）

## 05｜第四章：Agent 与人机结合（先别急着做）

### 原始要点

- 当团队（多 Agent）养好后，下一步是 **团队磨合**
- 常见问题：
  - 大家在 1、2、3 章花了大量时间
  - 结果“养了一个龙虾很酷炫”，但没真正帮到自己，甚至成本很高
- **根因往往在：需求梳理不清楚**
- 因此需要一套梳理需求的方法论：**“双 S SOP”理论**
  - **SOP 1**：龙虾（Agent）自己的 SOP
  - **SOP 2**：人机结合的 SOP
- 建议：
  - 如果能沉下心、不急着动手，会更划算
  - 因为当下技术迭代很快，你现在花大力气解决的问题，未来可能自动消失
  - 所以在“技术还不稳定”的阶段，**不推荐急着做第 4 章（人机结合）**

### 互联网补充 1：Agent Loop 视角——“人机结合”其实是流水线设计

OpenClaw 的 Agent Loop 文档描述了一个完整的 agentic loop：

- intake → context assembly → model inference → tool execution → streaming replies → persistence

你在“人机结合”阶段真正要做的事是把任务拆成 **可控的流水线**：

- 哪一步由人决策（目标、验收、风险边界）
- 哪一步交给 agent 执行（检索、生成、改写、整理、调用工具）
- 哪一步要检查/回滚（质量门、合规门、发布门）

> 引用：
> - https://docs.openclaw.ai/concepts/agent-loop.md

### 互联网补充 2：什么时候该“委派给 AI”？（管理学视角）

Ethan Mollick（One Useful Thing）提出了一个很实用的委派决策模型：委派是否划算，取决于三件事：

- **Human Baseline Time**：你自己做要多久
- **Probability of Success**：AI 一次做对的概率
- **AI Process Time**：你写指令 + 等待 + 评估输出要多久

直觉解释：如果你检查 AI 的成本很高，而 AI 又经常翻车，那就不值得委派；反之，任务越长、AI 成功率越高，越值得把它交给 AI。（参考：Management as AI superpower）

> 引用：
> - https://www.oneusefulthing.org/p/management-as-ai-superpower

### “双 S SOP”建议写法（更落地）

- **Agent SOP（龙虾 SOP）**
  - 输入格式（必须字段）
  - 工具链（调用顺序）
  - 失败重试策略（最多重试 N 次）
  - 输出格式（交付物清单）
- **Human SOP（人机 SOP）**
  - 目标与范围（要/不要做什么）
  - 验收标准（什么叫好）
  - 风险点清单（哪些必须人工确认）
  - 发布流程（谁批准、何时发布、如何回滚）

## 06｜第五章节：第一波龙虾如何挣钱？（Skill 如何挣钱？）

### 原始要点

- 我盘点了身边 **三类**情况：这三类赚的是 **时间窗口的钱**
- 但更核心的是：
  - **长期钱**来自把 Agent 与自己的业务结合，持续放大价值

### 互联网补充 1：为什么“Skill 能挣钱”（因为它天然可被版本化/分发/复用）

OpenClaw 官方对 Skill 的定义是：一个 skill 是“版本化的 bundle”，包含 `SKILL.md` 以及相关文件；每次发布会产生新版本，注册中心保存版本历史，用户可审计并安装。（参考：ClawHub / Skills 文档）

这句话很关键：它把“skill”从一次性脚本，变成了**可复制交付的产品单元**。

> 引用：
> - ClawHub（skill registry + versioning + distribution）：https://docs.openclaw.ai/tools/clawhub
> - Skills（skill 目录 + 规范 + precedence）：https://docs.openclaw.ai/tools/skills.md

### 互联网补充 2：Skill 变现的 6 种更具体路径（可直接做一页）

下面 6 条是“围绕 skill 的真实可落地生意”，你可以按听众画像挑 2～3 个重点讲：

1. **Skill 代练 / 定制交付（服务）**
   - 交付物：帮客户把某个工作流跑通、把 skill 写成可复用的 `SKILL.md` + 脚本/模板
2. **Skill 包售卖（产品）**
   - 交付物：一组可组合的 skills（比如“自媒体增长包”“SaaS 运营包”），附带演示与文档
3. **Skill 订阅更新（持续收入）**
   - 交付物：每周/月更新新版本（适配平台规则变化、模型变化、接口变化）
   - 利用 ClawHub 的“版本化”特性，发布与迭代成本低
4. **Skill 驱动的交付型外包（结果付费）**
   - 对外卖结果：比如“每周产出 20 条小红书图文+排期+复盘”，skill 是你的生产引擎
5. **Skill 教育 / 训练营（知识付费）**
   - “教你把 skill 养出来”：提供模板、作业、点评与落地辅导
6. **Skill 上云（SaaS 化）**
   - 把 skill 的一部分能力变成 API/产品功能（见下一章 SaaS 结合）

> 备注：OpenClaw 官方并没有直接写“skill 怎么定价”，但它把 skill 定义为**可发布、可安装、可版本化**的标准件；而标准件就具备商品化基础。

### 互联网补充 3：从“代练”到“产品化”的常见路径（你原来的逻辑保留）

建议把变现分成两类，讲起来更清楚：

- **窗口钱（服务型）**：
  - 帮别人“代练” Agent / 搭工作流 / 定制 skill
  - 特点：交付快、现金流快，但依赖个人精力
- **长期钱（产品型）**：
  - 把可复用能力标准化成：模板、技能包、订阅服务、SaaS 功能
  - 特点：前期慢，但可复利与规模化

你也可以用上一章 Mollick 的模型来解释：当 AI 能把一个原本要 10 小时的任务变成 1 小时，并且成功率足够高时，用户愿意为“节省的时间”买单——这就构成了付费意愿的底层逻辑。

## 07｜第六个话题：SaaS 产品如何突围？（SaaS 如何和龙虾/OpenClaw 结合？）

### 原始要点

- 结合两类人群：
  - 做自媒体的小伙伴（可交流）
  - 做 SaaS 产品的小伙伴
- SaaS 与龙虾（Agent）结合的两点：
  1. **借大事 / 借小事**
     - 以我自己的产品为例：最早就是“借龙虾的事”
     - 关键：**要给龙虾做嫁衣**，先“去自己化”，别一开始就想着建自己的生态
  2. **做自己的生态：让自己的龙虾也多一个 channel**
- 优先级建议：
  - **先做第 1 点，再做第 2 点**
- 呼应第 1 章：
  - 龙虾之所以能火，一个重要原因就是它多了 **channel**
  - 给产品也配上 channel，是很好的破局之道

### 互联网补充 1：OpenClaw 的定位决定了“怎么结合”

OpenClaw 在 README/FAQ 里都反复强调：

- 它是运行在你设备上的 personal AI assistant
- **Gateway 是 always-on control plane**（控制面），而 assistant 才是产品

这句话翻译成 SaaS 语言就是：OpenClaw 更像一层“可编排的控制平面”，可以把你的 SaaS 接到用户常用的聊天入口上，让用户通过 channel 使用你服务的能力。

> 引用：
> - README：https://github.com/openclaw/openclaw
> - FAQ：https://docs.openclaw.ai/help/faq

### 互联网补充 2：SaaS × OpenClaw 的 5 种具体结合方式（从轻到重）

1. **把你的 SaaS 当作“一个 tool/skill”接入（最轻）**
   - 形式：写一个 skill，让 agent 能调用你的 SaaS API（查询/创建/更新/报表/导出）
   - 价值：用户直接在 Telegram/飞书/微信等入口完成操作
2. **把你的 SaaS 变成“工作流的中间站”（更贴近真实业务）**
   - 形式：OpenClaw 负责“收集需求→拆解→调用工具”，SaaS 负责“存储/协作/审批/可视化”
3. **把 OpenClaw 变成你的“多渠道前台”（channel-first）**
   - 形式：你不先做复杂 UI，而是先用 channel 做 MVP：用户发消息 → agent 帮他完成任务 → 回传结果
   - 适用：早期验证、获客、提高留存
4. **用 OpenClaw 的 Gateway HTTP 能力做“可控的自动化入口”**
   - OpenClaw Gateway 暴露 `POST /tools/invoke`，允许通过 HTTP 直接调用单个 tool（仍受 gateway auth 与 tool policy 约束）
   - 你可以把它当作“受控的自动化执行器”：从你的 SaaS 服务器触发某个工具执行（例如批量抓取、生成、同步）
   - 引用：Tools Invoke API：https://docs.openclaw.ai/gateway/tools-invoke-http-api.md
5. **做一体化：把你自己的“channel + gateway + agent”打包成 SaaS（最重）**
   - 这对应你原文提到的“做自己的生态，让自己的龙虾也多一个 channel”

### 互联网补充 3：为什么“channel 是破局点”（更贴近 OpenClaw 特色）

OpenClaw 的 Features/Architecture 强调：

- 多 channel 是“使用频率”的放大器（用户在聊天里更高频触达）
- gateway 作为单点真相（single source of truth）负责路由、会话与工具调用

把这翻译成 SaaS 语言：

- **先借势已有 channel**：用现成渠道把用户拉到你这套工作流里（低摩擦）
- **再做自己的 channel**：当你有了稳定工作流与留存理由，再把入口做进产品（更强粘性）

> 引用：
> - Features：https://docs.openclaw.ai/concepts/features.md
> - Gateway Architecture：https://docs.openclaw.ai/concepts/architecture.md

## 08｜企业为什么要更早抓紧龙虾？（2B 总论）

> 这一章的目标：把“龙虾/Agent 很酷”翻译成企业能听懂的 **ROI + 风险边界 + 落地路线图**。

### 08.1 企业早抓的本质：先抢“入口”和“流程主导权”

企业做 AI，不是为了多一个聊天机器人，而是为了更快地把关键流程变成：

- **随时可触达（channel）**：员工/客户用消息入口触发
- **可编排（control plane）**：把“信息 → 决策 → 执行 → 回传”串起来
- **可审计（governance）**：能追溯、能限权、能回滚

OpenClaw 的官方表述非常接近这个企业解释：它强调 **Gateway 是 always-on control plane（控制面）**，assistant 才是产品；并且能接入你已经在用的消息面（messaging surfaces）。

> 引用：README / FAQ
> - https://github.com/openclaw/openclaw
> - https://docs.openclaw.ai/help/faq

### 08.2 企业落地最小闭环（推荐 2 周内做出来）

不要从“做一个万能机器人”开始，企业落地建议按最小闭环：

1. **选一个高频入口**：飞书 / Slack / WhatsApp / Telegram（你们真实在用的）
2. **选一个高频场景**（建议从“纯文本+低风险”起）：
   - 客服/工单分流与总结
   - 销售线索分拣与跟进提醒
   - 运营日报/周报自动生成（汇总各系统数据）
3. **设一个硬指标**：
   - 首响时间（FRT）
   - 人均处理单量
   - 跟进完成率
   - 报表产出耗时

> 你讲课时可以用一句话总结：**一个入口 + 一个场景 + 一个指标 = 企业第一波 AI 胜利。**

### 08.3 风险与治理：企业敢用的关键，是“默认安全”

企业更早抓紧龙虾的前提是安全边界清晰。OpenClaw 的安全文档强调它的信任模型是“个人助手/单信任边界”，如果要多人/多团队使用，需要拆分信任边界（多网关/多账号/多隔离）并做权限治理。

同时它提供了可操作的安全审计命令（`openclaw security audit`）帮助排雷。

> 引用：Security
> - https://docs.openclaw.ai/gateway/security/index.md

### 08.4 企业“可复制”的组织打法：多 Agent = 多岗位

你可以把“龙虾团队”讲成企业最熟悉的结构：

- **SalesOps Agent**：线索分拣、跟进、邮件/话术草拟
- **SupportOps Agent**：工单分类、回复草拟、升级/关闭建议
- **GrowthOps Agent**：内容日历、渠道分发、数据复盘
- **Dev/IT Agent**：技能开发、系统对接、自动化

OpenClaw 的 Multi-Agent Routing 文档强调：多 agent 是“隔离的 brain”（独立 workspace/state/sessions），这正是企业“岗位隔离”的工程基础。

> 引用：Multi-Agent Routing
> - https://docs.openclaw.ai/concepts/multi-agent.md

## 09｜SaaS 企业：怎么和龙虾/OpenClaw 结合（实战 + 效果）

### 09.1 典型 SaaS 痛点（高频且可量化）

- 线索多但无效：没人分拣、没人快速跟进
- 客服压力大：FAQ 重复、工单总结慢、升级不及时
- 运营复盘慢：数据分散在多个系统，周报要人肉拼
- 交付与续费靠人：客户健康度不可见、流失预警晚

### 09.2 结合方式（按落地难度从轻到重）

1. **把你的 SaaS API 封装成 skill（最快）**
   - 让 agent 能“查客户/查订单/查用量/开工单/更新状态/导出报表”
2. **channel-first 做 MVP（获客最快）**
   - 用户在飞书/Telegram 里发一句话 → agent 通过你的 API 完成任务 → 回传结果
3. **把 OpenClaw 当“自动化执行器”**
   - 你的 SaaS 服务器通过 `POST /tools/invoke` 触发某个工具执行（仍受网关鉴权+工具策略限制）

> 引用：Tools Invoke API
> - https://docs.openclaw.ai/gateway/tools-invoke-http-api.md

### 09.3 3 个 SaaS 实战剧本（可直接照做）

**剧本 A｜线索分拣 + 跟进提醒（SalesOps）**
- 输入：渠道私信/表单/邮件线索（文本即可）
- 龙虾做：提取意向点 → 打标签（行业/预算/紧急度）→ 生成 2 版跟进话术 → 设定跟进节奏
- 人做：最终确认发送、关键客户人工跟进
- **效果指标**：首响时间、跟进完成率、有效线索占比

**剧本 B｜客服工单“分流-草拟-总结”（SupportOps）**
- 输入：客户聊天记录/截图/语音
- 龙虾做：问题归类 → 生成回复草稿（含必要信息收集）→ 输出工单总结与下一步
- 人做：审核发送、必要时升级
- 可选：开启 inbound media understanding，把语音/图片先摘要成文本再走流程

> 引用：Media Understanding（入站音视频/图片预摘要）
> - https://docs.openclaw.ai/nodes/media-understanding

**剧本 C｜运营日报/周报自动化（GrowthOps）**
- 输入：多个系统的零散数据、会议纪要、渠道反馈
- 龙虾做：每天定时拉取/汇总 → 输出“结论 + 异常 + 建议动作”
- 实现：用 Gateway 的 cron job 让它每天固定时间跑一次

> 引用：Cron jobs
> - https://docs.openclaw.ai/automation/cron-jobs.md

### 09.4 SaaS 预期效果怎么讲（别用空话，用指标）

你可以把“效果”讲成三类可量化变化（不承诺具体数值，承诺测量方法）：

- **速度**：首响时间、报价/工单处理时长、报表产出时长
- **产能**：人均处理单量、人均跟进数、内容产出频率
- **质量**：一次解决率、升级率、重复问题占比、客户满意度

## 10｜外贸电商：询盘到订单的龙虾打法（实战 + 效果）

### 10.1 外贸电商的“高价值环节”在哪里

- 询盘（RFQ）响应速度：晚 2 小时可能就丢单
- 多语言沟通与信息收集：参数、交期、付款、物流
- 报价与跟进：报价单/PI/条款说明 + 持续 follow-up
- 资料与合规：装箱单、发票、报关材料（先从文本模板自动化开始）

### 10.2 实战工作流（建议先从 WhatsApp/邮件线索开始）

**工作流｜询盘自动结构化 → 报价草拟 → 跟进节奏**
1. 客户发来询盘（文本/语音/截图都可能）
2. 龙虾先做“结构化字段”：产品、数量、规格、目标价、交期、目的港、认证要求
3. 输出 2 份结果：
   - 给销售：一页“关键信息 + 缺失问题清单”
   - 给客户：一版“确认问题 + 下一步”英文/目标语种草稿
4. 报价单/PI：用模板填充（先让龙虾产出表格/Markdown，再落到你们现有系统）
5. 跟进：用 cron job 设置 24h/72h follow-up 提醒

> 引用：Cron jobs
> - https://docs.openclaw.ai/automation/cron-jobs.md

### 10.3 预期效果（建议你现场用这 4 个指标讲）

- **FRT**：询盘首响时间（分钟/小时）
- **Quote TAT**：从询盘到可发报价的耗时
- **Follow-up 完成率**：该跟的是否都跟了
- **信息缺失率**：第一次回复能否把关键字段收齐（少来回=更快成交）

## 11｜独立站品牌：内容-投放-客服-复盘一体化（实战 + 效果）

### 11.1 独立站品牌最“烧钱”的 3 件事

- 付费投放：素材与文案迭代慢
- 内容与 SEO：持续产出难，站内页面更新慢
- 客服与口碑：重复问题多，售后拖慢复购

### 11.2 实战剧本（独立站最适合用“多 Agent 分工”跑流水线）

**剧本 A｜广告素材与文案工厂（GrowthOps + CreativeOps）**
- 输入：产品卖点、用户评价、竞品对比
- 龙虾产出：
  - 10 个广告角度（痛点/人群/场景）
  - 每个角度 3 条短文案 + 1 条长文案
  - UGC 口播脚本（15s/30s/60s）
- 人做：选角度、投放测试
- **效果指标**：素材迭代频率、测试轮数、CTR/CVR 的提升速度

**剧本 B｜客服知识库 + 回复宏（SupportOps）**
- 输入：历史客服对话/差评/退货原因
- 龙虾产出：FAQ 分类、标准回复宏、升级条件（什么时候必须人工介入）
- **效果指标**：一次解决率、人工介入比例、响应时长

**剧本 C｜周度复盘（GrowthOps）**
- 输出：一页“本周增长结论 + 三个实验建议 + 下周内容/投放计划”
- 用 cron job 固定每周生成，减少“复盘拖延”

> 引用：Cron jobs
> - https://docs.openclaw.ai/automation/cron-jobs.md

## 12｜MCN 机构：选题-脚本-制作-分发的龙虾流水线（实战 + 效果）

### 12.1 MCN 的瓶颈不是创意，是“周转率”

MCN 赚钱靠稳定产出与爆款概率，但日常最卡的往往是：

- 选题靠感觉，不可复用
- 脚本与口播来回改，版本混乱
- 一条内容要多平台改写，重复劳动
- 复盘不成体系，下一次还踩同样坑

### 12.2 实战组织结构（一个 MCN = 一个多 Agent 小团队）

- **Producer Agent**：选题库、选题评审、拍摄清单
- **Script Agent**：脚本/标题/钩子、口播稿、分镜
- **Editing Agent**：金句点位、字幕要点、封面文案（可接你们剪辑工具链）
- **Ops Agent**：多平台分发排期、数据汇总、复盘

### 12.3 实战工作流（你可以把它讲成 SOP）

1. 每天固定时间跑“热榜/竞品监测 → 选题候选”
2. 选题入库（标题、目标人群、核心冲突、3 个钩子）
3. 脚本生成（15s/30s/60s 版本）+ 分镜（可选）
4. 发布后自动复盘：把数据汇总成“结论 + 下一轮动作”

实现上：
- 用 cron jobs 定时触发（选题/复盘）
- 用 hooks 做事件触发（比如发 `/new` 或某个命令时自动写入复盘文件）

> 引用：
> - Cron jobs：https://docs.openclaw.ai/automation/cron-jobs.md
> - Hooks：https://docs.openclaw.ai/automation/hooks.md

### 12.4 预期效果（MCN 讲这 3 个就够）

- **内容周转率**：从选题到发布的平均周期
- **复用率**：脚本/标题/模板被复用的次数
- **产出稳定性**：每周稳定发布数量（爆款概率=基数×质量）

- **链接**：https://cloud.tencent.com/developer/article/2628308
- **画像**：想“偶尔放纵又不想胖”的个人用户；对优惠券/活动敏感。
- **一句话价值**：一句“帮我领个券”，AI 能自动领到优惠券并给出控热量的点餐组合建议。
- **可复制 SOP**：
  1) 安装 `mcd` Skill
  2) 登录麦当劳 MCP 平台生成 `token`
  3) 在聊天通道发送 `/new` 激活会话
  4) 对话示例：`帮我领个券` / `最近有什么活动？` / `我想吃板烧，帮我组合一下让我能吃饱，顺便控热量`
- **交付物**：优惠券领取结果、活动清单、两套套餐方案（含卡路里标注）。
- **指标建议**：
  - 领取优惠券张数（文中提到一次领到 5 张）
  - 选餐决策时间（从“翻来翻去”降到 1-2 轮对话）
- **踩坑/注意**：涉及账号/凭证（token），必须强调“只在可信环境提供、最小权限、用完即换”。

#### A.1.2 10 分钟打造 AI 选股系统（实时行情 API + 调度）
- **来源**：腾讯云开发者社区
- **链接**：https://cloud.tencent.com/developer/article/2631336
- **画像**：信息过载的散户/副业投资者；希望“过滤垃圾信息、直接给结论”。
- **一句话价值**：结合实时行情 API 与自动化调度，个人也能快速搭建“专属股票分析系统”。
- **可复制 SOP**：
  1) 明确输入：关注标的/行业、风险偏好、更新频率
  2) 接入实时行情 API（按文章方案）
  3) 设定每日/每小时调度：拉取行情→筛选→输出结论
  4) 输出格式固定：`今日关注清单 + 触发原因 + 风险提示 + 明日观察点`
- **交付物**：手机端可查看的最新行情与筛选结果（文章描述）。
- **指标建议**：
  - 每日节省信息筛选时间
  - 关注标的命中率（用回测/人工记录，不承诺收益）
- **踩坑/注意**：课程里必须加“仅供学习，不构成投资建议；避免用高收益噱头”。

#### A.3.1 OpenClaw 接入飞书云文档/权限/导出：把机器人变成“文档操作员”
- **来源**：腾讯云开发者社区
- **链接**：https://cloud.tencent.com/developer/article/2631667
- **画像**：企业/团队使用飞书文档、表格、Wiki；需要自动生成文档、分配权限、导出文件。
- **一句话价值**：给飞书机器人开通 docx/docs 等权限后，OpenClaw 可以创建/编辑文档、管理权限、导出 PDF。
- **可复制 SOP**：
  1) 飞书开放平台 → 自建应用 → 权限管理 → 开通云文档相关权限（文中建议同时开 `docx` + `docs`）
  2) 回到飞书对话：`创建一个文档，介绍一下飞书云文档是什么`（示例提示词在文中）
  3) 权限不够时：`这个文档我好像没有编辑权限，给我授权一下`
  4) 需要交付文件：`帮我把这个文档导出为PDF，然后发给我吧`
- **交付物**：文档链接、权限设置结果、导出的 PDF。
- **指标建议**：
  - 文档产出时间（从 30-60 分钟 → 3-5 分钟）
  - 权限/导出操作减少的人工步骤数
- **踩坑/注意**：权限配置容易过度授权；建议按“最小权限”开通，重要动作加人工确认。
